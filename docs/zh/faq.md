# 常见问题 (FAQ)

## 通用问题

### Q: 助记词丢失了怎么办？

::: danger 无法恢复
助记词是恢复同步链的唯一凭证，丢失后无法找回，需要创建新的同步链。
:::

**预防措施：**
1. 创建时立即备份到多个安全位置
2. 使用密码管理器存储
3. 手写纸质备份

### Q: 可以同时登录多少设备？

**答案：** 无限制。

Note Sync Now 采用零账户设计，任何持有助记词的设备都可以加入同步链。但需要注意：
- 单房间建议不超过 100 设备
- 设备数量影响同步性能

### Q: 同步延迟是多少？

**典型延迟：**

| 网络条件 | 延迟范围 |
|---------|---------|
| 本地网络 | < 50ms |
| 同地区 | 50-150ms |
| 跨地区 | 150-500ms |
| 跨洲际 | 500-1000ms |

### Q: 数据存储在哪里？

**存储架构：**

| 位置 | 存储内容 | 加密状态 |
|------|---------|---------|
| 客户端 IndexedDB | 笔记内容、历史 | 明文（本地） |
| 服务端 Redis/SQLite | 加密数据 | 密文 |
| 服务端内存 | 加密数据 | 密文 |

::: tip 零信任架构
服务端永不接触明文，所有加密在客户端完成。
:::

## 加密相关

### Q: 为什么选择 AES-256-GCM？

**答案：**

1. **安全性**：NIST 认证，军用级加密强度
2. **认证加密**：同时提供机密性和完整性
3. **性能**：硬件加速支持，适合大文件
4. **标准化**：广泛审计，无后门风险

### Q: PBKDF2 迭代次数为什么是 100,000？

**答案：**

```
破解成本估算：
- 单次计算：~1ms
- 100,000 次迭代：~100ms
- 尝试 1 亿个助记词：~115 天（单机）
- GPU 加速：仍需数周
```

迭代次数是性能与安全的平衡点，未来可随硬件升级而增加。

### Q: 助记词有多安全？

**安全强度：**

| 属性 | 值 | 说明 |
|------|---|------|
| 熵值 | 128 bits | 2^128 种可能组合 |
| 词表大小 | 2048 词 | BIP39 标准 |
| 暴力破解 | 不可行 | 超出当前计算能力 |

## 同步相关

### Q: 冲突如何解决？

**三路合并算法：**

```
基础版本: "Hello"
本地版本: "Hello World"
远端版本: "Hello!"

合并结果: "Hello World!" (无冲突)
         或标记冲突区域 (有冲突)
```

### Q: 离线时编辑的内容会丢失吗？

**不会。** 离线编辑会存储在 IndexedDB 中，网络恢复后自动同步。

限制：
- 最多缓存 100 条离线更新
- 同一笔记只保留最新版本

### Q: 大文件如何处理？

**分块传输机制：**

| 大小 | 处理方式 |
|------|---------|
| < 100 KB | 直接传输 |
| 100 KB - 5 MB | 自动分块传输 |
| > 5 MB | 拒绝传输 |

## 部署相关

### Q: 推荐什么部署方式？

| 场景 | 推荐方案 |
|------|---------|
| 个人使用 | Docker 单容器 |
| 团队使用 | Docker Compose + Redis |
| 生产环境 | Kubernetes + Redis Cluster |

### Q: Redis 和 SQLite 如何选择？

| 对比项 | Redis | SQLite |
|--------|-------|--------|
| 性能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 持久化 | 可配置 | 原生 |
| 部署复杂度 | 中等 | 简单 |
| 水平扩展 | 支持 | 不支持 |
| 适用场景 | 高并发生产 | 小规模部署 |

### Q: 如何备份数据？

**服务端备份：**

```bash
# Redis 备份
redis-cli BGSAVE
cp /var/lib/redis/dump.rdb /backup/

# SQLite 备份
cp data/sync.db /backup/sync-$(date +%Y%m%d).db
```

**客户端备份：**
- 导出笔记为 Markdown/文本文件
- 保存助记词到安全位置

## 安全相关

### Q: 服务端可以看到我的笔记吗？

**不能。** 端到端加密设计确保：

1. 加密在客户端完成
2. 服务端只存储密文
3. 服务端没有解密密钥

### Q: 如果服务端被攻击怎么办？

**影响分析：**

| 攻击类型 | 影响 | 保护措施 |
|---------|------|---------|
| 数据泄露 | 无 | 数据已加密 |
| 服务中断 | 暂时无法同步 | 离线功能正常 |
| 代码篡改 | 需要用户信任 | 开源可审计 |

### Q: 如何确保客户端代码安全？

**验证方法：**

1. **源码审计**：项目完全开源
2. **自行构建**：从源码构建 Docker 镜像
3. **网络分析**：检查 WebSocket 流量（密文）
